## 1. Введение и постановка задачи

Задача — построить end-to-end ASR-систему распознавания произнесённых на русском языке чисел. Основные ограничения:

- модель обучается **с нуля**, без использования предобученных весов;
- количество параметров — **не более 5 млн**;
- вход — моно-аудио 16 кГц.

Метрика — **гармоническое среднее CER** между in-domain и out-of-domain спикерами:

$$
\text{HarmonicCER} = \frac{2 \cdot \text{CER}_{in} \cdot \text{CER}_{out}}{\text{CER}_{in} + \text{CER}_{out}}
$$

Такая метрика штрафует модели, которые «выучили» только знакомых спикеров и проседают на новых — то есть напрямую измеряет генерализацию.

В проекте я реализовал базовое решение и инфраструктуру для удобной работы над дальнейшим улучшением результата. Григорий же занимался итеративным улучшением этого бэйзлайна.

Основной код — в `group-projects/gp1/src/`, baseline - `baseline.ipynb`, обучение запускается из `pipeline.ipynb`, инференс — из `kaggle_public.ipynb`. Полное дерево репозитория — в §7.

## 2. Данные

- **Train:** 12 553 записи, 6 спикеров (A–F), частоты 22–24 кГц → ресемпл до 16 кГц.
- **Dev:** 2 265 записей, 10 спикеров, 16 кГц. Из них 6 пересекаются с трейном (in-domain), 4 новых (out-of-domain — H, I, J, K).
- **Test (Kaggle):** 2 582 файла, смесь wav/mp3, без меток.
- **MUSAN `noise/`** — внешние шумы для аугментации данных

Все аудио приводятся к 16 кГц, моно; далее считается лог-мел-спектрограмма (80 банок, `n_fft=256`, `hop=160`) и нормируется по каждому сэмплу.

## 3. Baseline

### Подготовка инфраструктуры

#### Датасет

Для удобства дальнейшей работы был реализовал `ASRDataset`, который:

- Инкапсулирует логику для работы с данными;
- Предзагружает аудио, что ускоряет обучение;

И дальнейшие DataLoader'ы, которые подгружают нужные датасеты.

#### Аугментация

Также подготовлен базовый `Augmenter`, который позволяет применять следующие аугментации: гауссовый шум, ускорение/замедление звуковой дорожки, изменение громкости, временной сдвиг, клиппинг, эффект эхо.

#### Нормализация

Подготовлена логика для унифицированной нормализации данных. Таким образом, при обучении можно свободно переключать их между собой не затрагивая остальные процессы. Были реализованы:

- `RussianAlphabetWordNormalizer` - нормализация на уровень букв;
- `DigitCharNormalizer` - нормализация на уровень цифр;
- `RussianWordNormalizer` - номализация на уровень цифр, но в словесном эквиваленте (практически аналогичен предыдущему);
- `UnspittableNormalizer` - нормализация в неделимые слова, которые произносит человек во всех аудио для чисел от 1000 до 999999. Именно на нем, удалось добиться наилучшего результата.

#### Обучение

Далее реализован кастомный PyTorch Lightning Trainer для удобного обучения всех последующих моделей с возможностью логирования всех результатов обучения в требуемом формате совместно с используемыми параметрами для последующего сравнения.

### Baseline метрики

В качестве модели была выбрана связка CNN + BiLSTM-энкодер (~4.8 М параметров) с CTC-лоссом. Здесь удалось добиться результата **17.24 %** гармонического CER (inD 16.7 %, ooD 17.8 %, train ~0.9 % — слабое переобучение) с использованием `UnspittableNormalizer` и базовыми настройками аугментаций. Таким образом, было протестировано, что весь базовый pipeline работает со всеми нормализаторами (на других при той же архитектуре получались значения 40% (без аугментаций) => 34% => 27% (`RussianAlphabetWordNormalizer` + аугментации)) и аугментациями. Далее можно переходить к оптимизации архитектуры модели и более глубокой работе с данными.

## 4. Сводка по итерациям

На каждой итерации вводилось ровно одно существенное изменение. Все эксперименты логировались в `logs/<experiment>/`.

| Шаг | Изменение | Harmonic CER | Относит. улучш. |
|-----|-----------|--------------|-----------------|
| Baseline | CNN + BiLSTM | 17.24 % | — |
| v0.1 | Корректные `input_lengths` в CTC, валидация по сплитам | 19.96 % | ухудшение* |
| v0.2 | + SpecAugment (freq/time masks) | 11.59 % | −42 % |
| v0.3 | + MUSAN real-noise mixing (SNR 5–25 дБ) | 6.03 % | −48 % |
| v0.4 | + `UnspittableNormalizer` (позиционно-цифровые токены вместо char-level) | 3.99 % | −34 % |
| v0.5 | + `ModelCheckpoint(monitor="val/harmonic_cer")` — отбор лучшей эпохи | 3.24 % | −19 % |
| **v0.6** | **Conformer encoder (2.5 М параметров) + OneCycleLR warmup** | **0.75 %** | **−77 %** |

\* v0.1 показала более высокий результат из-за того, что в baseline CTC видел паддинговые фреймы и считал «лёгкое» решение. После фикса результат стал немного честнее.

**Итог:** `17.24 % → 0.75 %` — снижение в **23 раза**.

## 10. Выводы

- **Работа с данными > современная архитектура** на малых датасетах. Из 23× суммарного снижения CER на долю архитектурной замены (Conformer) пришлось около 4× (3.24 % -> 0.75 %), остальное — feature engineering, аугментация, грамотный чекпоинтинг и исправление ошибок baseline'а
- **CTC — чувствительный к длинам формат** Любая небрежность с `input_lengths` или с паддингом выходных фреймов сразу проявляется в виде инфляции CER (или его занижения)
- **Conformer требует warmup** Без него модель расходится даже на сходящихся гиперпараметрах
- **UnspittableNormalizer** — креативный подход к токенизации дает существенный прирост в качестве
